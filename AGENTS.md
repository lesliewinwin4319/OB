# OB Project — AI Agent Context

## What is OB?

OB is an iOS social app built around **anti-selfie** photography. Users can only post photos taken by others — you cannot post photos you took yourself. The app is being built by a small team with AI agents handling development.

## Server Stack (`ob-server/` and `server/`)

| Layer | Choice |
|---|---|
| Framework | NestJS (v11) |
| Language | TypeScript |
| ORM | TypeORM (v0.3) |
| Database | PostgreSQL (Railway-managed) |
| Cache | Redis (Railway-managed, via ioredis) |
| Auth | WeChat OAuth code exchange → JWT |
| Deployment | Railway (Docker, auto-migration on startup) |
| ID generation | Snowflake (custom pure function, configurable MACHINE_ID) |

### Auth Flow

1. iOS sends WeChat `code` to `POST /api/v1/auth/login`
2. Server exchanges code with WeChat API (or mock in dev) → gets `openid`
3. Lookup or create user by `openid`
4. Sign JWT containing `{ sub: userId, status: UserStatus }`
5. Return `{ token: "..." }` — field name is **`token`**, not `accessToken`

### JWT & Guards

- **JwtStrategy** — queries DB on every request, attaches full user object to `req.user`
- **JwtAuthGuard** — validates JWT signature
- **ProfileCompleteGuard** — checks `status` field inside the JWT payload (NOT a DB lookup). Blocks `PENDING_PROFILE` users from protected routes.

### User Status Enum

```typescript
enum UserStatus {
  PENDING_PROFILE = 'PENDING_PROFILE',  // registered, profile not yet completed
  ACTIVE = 'ACTIVE',                    // normal user
}
```

### User Entity Key Fields

- `id` — Snowflake bigint string (generated by app, not DB)
- `openid` — WeChat openid, unique
- `nickname` — set during profile completion
- `avatarUrl` — optional, nullable
- `status` — `UserStatus` enum
- `deletedAt` — soft delete (TypeORM `@DeleteDateColumn`)
- `updatedAt` — auto-updated via DB trigger

### Mock Mode (Development)

`WechatService` has a mock mode: **any non-empty string** passed as `code` is accepted and returns a deterministic fake `openid`. Controlled by env var `WECHAT_MOCK=true`.

## iOS Stack (`OB/`)

| Layer | Choice |
|---|---|
| Language | Swift 5.9+ |
| UI | SwiftUI |
| Project config | XcodeGen (`project.yml`) |
| Auth token storage | Keychain |
| Networking | URLSession (no third-party HTTP lib) |

### iOS Auth Conventions

- JWT stored in Keychain (not UserDefaults)
- Cold-start: app reads token from Keychain → validates with server before entering main flow
- API base URL configured via `Config.swift` or equivalent (not hardcoded)
- Server response for login: `{ "token": "..." }` — parse field name `token`

---

## API Conventions

- Base path: `/api/v1/`
- Auth header: `Authorization: Bearer <token>`
- All responses follow: `{ data: ..., message: "..." }` or NestJS default error shape
- Health check endpoint: `GET /api/v1/health`

## Code Style Rules

- **No speculative abstractions** — implement exactly what the spec asks
- **No extra comments or docstrings** on code you didn't have reason to explain
- **No backwards-compat shims** — change the code directly
- **No mocks in integration tests** — use real DB connections
- TypeScript strict mode; no `any` unless unavoidable
- NestJS modules must be explicitly imported — no implicit global registration
- Migration files go in `src/database/migrations/` and must be reversible (`down()` implemented)
