# OB 项目分支策略

## 分支结构

```
main          ← 稳定版，只接受来自 dev 的合并，不直接提交代码
dev           ← 日常开发主线，功能验证通过后汇入
feature/xxx   ← 单个功能开发分支，从 dev 拉出，完成后合并回 dev
hotfix/xxx    ← 线上紧急修复，从 main 拉出，修完同时合并回 main 和 dev
```

## 标准开发流程

```
1. 新功能启动
   Adam 从 dev 创建 feature 分支
   命名规范：feature/模块名，如 feature/photo-capture

2. 开发阶段
   Louis / Seth 在 feature 分支上开发
   每个功能模块独立分支，互不干扰

3. 测试阶段
   Rock 在 feature 分支上执行测试用例
   发现 Bug → 在同一 feature 分支修复 → 回归

4. 合并到 dev
   所有测试用例通过 → Adam 合并 feature → dev
   删除已合并的 feature 分支

5. 合并到 main（阶段性上线）
   dev 稳定后 → Adam 合并 dev → main → 推 GitHub
```

## 紧急修复流程

```
线上发现 Bug
→ Adam 从 main 拉 hotfix/xxx 分支
→ Seth/Louis 修复
→ Rock 验证
→ 合并回 main + dev
→ 删除 hotfix 分支
```

## 各角色职责

| 角色 | 分支职责 |
|------|----------|
| Adam | 创建/合并/删除分支，解决冲突，管理 main 和 dev |
| Louis | 在 feature 分支上开发 iOS 代码 |
| Seth | 在 feature 分支上开发后端代码 |
| Rock | 在 feature 分支上执行测试，通过后通知 Adam 合并 |
| 用户 | 确认合并到 main 的时机，审阅更新日志后授权推送 |
