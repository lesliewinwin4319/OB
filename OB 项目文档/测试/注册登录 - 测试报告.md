## 3. Bug 详情

### BUG-LOGIN-001

- **Bug ID**：BUG-LOGIN-001
- **严重等级**：Minor
- **所属模块**：服务端（Seth）
- **关联用例**：TC-044
- **复现步骤**：
  1. 用 POST /auth/wechat/login（mock code）创建 PENDING_PROFILE 用户，获取 Token
  2. 用该 Token 请求 GET /users/me/friend-requests
- **预期结果**：HTTP 403，响应体 errorCode=PROFILE_INCOMPLETE
- **实际结果**：HTTP 403，响应体 errorCode=FORBIDDEN，message=「请先完善个人资料」
- **影响**：客户端若依赖 errorCode 区分「未登录（401）」与「资料未填写（403 PROFILE_INCOMPLETE）」，将无法精确处理后者，可能展示通用错误文案而非引导用户完善资料

---

### BUG-LOGIN-002

- **Bug ID**：BUG-LOGIN-002
- **严重等级**：Major
- **所属模块**：服务端（Seth）
- **关联用例**：TC-019、TC-048
- **复现步骤**：
  1. 创建新用户，获取 Token
  2. 调用 POST /users/me/profile，传入 nickname=「😊😊😊😊😊😊😊😊😊😊😊」（11 个 Emoji，Swift Unicode scalar 计为 11，在客户端 20 字符上限内）
- **预期结果**：服务端接受，返回 200，用户状态变为 ACTIVE
- **实际结果**：服务端返回 HTTP 400「昵称不能超过 20 个字符」
- **根本原因**：服务端 users.service.ts 使用 JavaScript 原生 `string.length` 计算长度（surrogate pair 模式，BMP 外字符算 2），而 Swift 客户端使用 Unicode scalar 计数（每个码位算 1）。单个普通 Emoji（如 😊，U+1F60A）在 JS 中 length=2，因此服务端实际上限为 10 个 Emoji，与客户端的 20 个 Emoji 不一致。
- **影响**：用户在客户端输入 11-20 个 Emoji 昵称时，客户端不拦截（认为合法），提交后服务端拒绝，造成注册流程失败。组合 Emoji（如家庭 Emoji 👨‍👩‍👦，JS length=8）问题更严重，2-3 个即超限。
- **修复建议**：服务端改用 `[...nickname].length`（Spread 运算符按 Unicode codepoint 计数）或 `Array.from(nickname).length` 替代 `nickname.length`，与 Swift 的 `String.count` 对齐

### BUG-LOGIN-003

- **Bug ID**：BUG-LOGIN-003
- **严重等级**：Major
- **所属模块**：服务端（Seth）
- **关联用例**：TC-042
- **复现步骤**：
  1. 同时并发发送 2 个 POST /auth/wechat/login 请求，code 相同（如 mock_tc042_real）
- **预期结果**：两个请求均返回 200，携带相同 uid；或一个 200、一个 429（限流）；DB 只有 1 条记录
- **实际结果**：一个请求返回 200（正常），另一个返回 HTTP 500，errorCode=INTERNALSERVERERROR，message=「登录请求过于频繁，请稍后重试」；DB 确实只有 1 条记录（唯一性约束未被破坏）
- **影响**：
  1. statusCode=500 语义错误（这是业务限流，应为 429 Too Many Requests）
  2. 客户端收到 500 会触发通用错误处理（可能展示「服务器异常」而非「请稍后重试」），体验差
  3. 暴露了内部实现细节（限流机制）
- **修复建议**：将并发重复登录的响应改为 429 或直接 UPSERT 返回 200（推荐后者，登录本身应幂等）
