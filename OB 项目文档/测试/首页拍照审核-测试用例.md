## 0. 测试前环境检查（每次必做）

| # | 检查项 | 操作 | 通过标准 |
|---|--------|------|----------|
| 1 | App 已安装最新 build | Xcode 编译安装到真机 | 无编译错误，App 正常启动 |
| 2 | 相机权限已授权 | 系统设置确认 OB 相机权限为「允许」 | 首页相机预览正常 |

> 本模块为纯客户端开发，不依赖后端服务，无需检查 NestJS 服务和网络环境。

---

## 1. Test Strategy Summary

测试重点在于**快门拍照的真实性与防抖机制**、**相册异步保存与权限解耦（不阻塞业务流）**、**拍照确认页的放弃二次确认与确定流转**、**上传选择页好友 Token 单选互斥逻辑与申请按钮状态联动**、**NavigationPath Pop to Root 路由一致性**、以及**侧滑返回禁用、断网 Toast、杀进程恢复**等异常场景。全部用例需真机手动执行。

## 3. Potential Risks

1. **快门防抖时序竞态**：isCapturing 标记在 CaptureDelegate 回调中重置，如果 AVCapturePhotoOutput 回调延迟（如低光环境拍照耗时较长），按钮禁用时间可能过长，用户感知为"快门卡死"。测试时需在低光环境下验证 TC-002。

2. **UIImage 通过 NavigationPath 传递的内存峰值**：原图可能 10MB+，在 Navigation 跳转时如果 SwiftUI 对路由枚举做值拷贝（Hashable），可能触发内存峰值。连续多轮拍照（TC-058）时尤其需要关注 Xcode Memory Graph。

3. **相册保存与确认页的异步竞态**：保存动作和页面跳转并行，如果相册权限弹窗（首次请求）覆盖了确认页，用户可能困惑当前操作上下文。需确认弹窗层级不影响确认页交互（TC-007）。

4. **Pop to Root 后相机预览恢复**：NavigationPath 清空后，HomeView 重新成为顶层视图，CaptureSession 需要正确恢复 running 状态。如果 session 在子页面期间被暂停但未正确恢复，可能出现回到首页后黑屏。

5. **selectedFriends 数组状态残留**：上传选择页使用 @State 管理 selectedFriends，Pop to Root 时 View 销毁应自动清理。但如果 ViewModel 被提升为 @StateObject 或 @ObservedObject 且生命周期跟随父 View，可能出现第二轮拍照进入选择页时仍有上轮选中状态。

6. **侧滑返回禁用的系统兼容性**：自定义禁用侧滑返回手势在不同 iOS 版本上实现方式可能不同。需在 iOS 16 和 iOS 17+ 上都验证 TC-018 和 TC-034。

7. **Token 选择框的 UI 溢出**：当前是单选场景问题不大，但如果好友昵称很长（如"这是一个超级长的昵称测试"），Token 显示可能超出选择框宽度导致布局异常。

## 4. Blocker 定义

以下情况属于阻塞级（P0），任意一项未通过则**不允许进入下一模块**：

| 编号 | Blocker 场景 |
|------|-------------|
| B-01 | 点击快门无法触发拍照，或拍照后不跳转确认页（核心流程入口断裂） |
| B-02 | 快门防抖失效，连续点击触发多次拍照或多次 push（页面栈错乱） |
| B-03 | 相册权限请求阻塞了业务流程（用户无法继续操作确认页） |
| B-04 | 确认页点击「确定」无法进入上传选择页（流程断裂） |
| B-05 | 放弃二次确认弹窗不弹出，或点击「是」后不回首页（放弃通道断裂） |
| B-06 | 好友选择无法触发，或选中后 Token 不出现（选人交互断裂） |
| B-07 | 申请按钮在选中好友后仍为禁用态（无法完成提交） |
| B-08 | 点击「申请」后不回首页，或页面残留在中间态（路由管理失效） |
| B-09 | 侧滑返回未禁用，可以滑回上一页（违反产品决策） |
| B-10 | 杀进程后冷启动不回首页，恢复到中间态页面（生命周期处理错误） |
| B-11 | 断网点击「申请」App 崩溃（无异常兜底） |
| B-12 | 连续多轮操作后 App 崩溃或严重卡顿（内存管理失败） |
