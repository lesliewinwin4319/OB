# 注册登录 - 客户端接入服务端测试用例

版本：v1.0 · 2026-04-11
负责人：Louis（iOS 客户端）
对应改动：OBAPIClient / KeychainHelper / AuthStateManager / RootView

---

## 测试范围说明

本轮测试验证客户端与生产服务端（Railway）的完整接入，重点覆盖：
1. baseURL 切换后网络请求能否正常到达生产环境
2. Token 改用 Keychain 存储后的读写和冷启动恢复
3. 冷启动时 `/api/v1/users/me` 验证 token 有效性的完整分支
4. RootView `.unknown` 状态的 Loading UI 及后续跳转

---

## TC-01：Mock 登录成功 → 新用户（PENDING_PROFILE）

**前置条件**：清空 App 数据（Keychain + UserDefaults），使用全新 mock code

**步骤**：
1. 打开 App，点击「微信登录」（Dev 模式触发 mock 登录）
2. 等待请求返回

**预期结果**：
- 请求命中 `https://ob-production-8afc.up.railway.app/api/v1/auth/wechat/login`，返回 200
- `LoginResponse.accessToken` 字段正确解析（服务端字段名为 `token`，CodingKeys 映射后能取到值）
- Token 写入 Keychain，不写入 UserDefaults
- `registrationStep` 转为 `.pendingProfile`，页面跳转至资料填写页

---

## TC-02：Mock 登录成功 → 老用户（ACTIVE）

**前置条件**：使用已完善资料的账号对应的 mock code（或手动在服务端将账号置为 ACTIVE）

**步骤**：
1. 清空本地 Keychain，保留 UserDefaults 中无 token 状态
2. 打开 App，触发 mock 登录

**预期结果**：
- 服务端返回 `status: "ACTIVE"`
- Token 写入 Keychain
- 页面直接跳转至主页（MainTabView）

---

## TC-03：冷启动 - Token 有效，用户为 ACTIVE

**前置条件**：Keychain 中存有有效 JWT Token，UserDefaults 中 `OB_RegistrationStep` = `active`

**步骤**：
1. 强退 App，重新打开

**预期结果**：
- RootView 初始展示 `.unknown` Loading 界面（不超过 0.5 秒）
- `syncStatusFromServer()` 调用 `GET /api/v1/users/me` 携带 Keychain 中的 token
- 服务端返回 200，`status: "ACTIVE"`
- 页面自动跳转到主页，无任何闪烁或登录页出现

---

## TC-04：冷启动 - Token 有效，用户为 PENDING_PROFILE

**前置条件**：Keychain 中存有有效 Token，用户 status 为 PENDING_PROFILE

**步骤**：
1. 强退 App，重新打开

**预期结果**：
- Loading 界面短暂展示
- `GET /api/v1/users/me` 返回 `status: "PENDING_PROFILE"`
- 跳转至资料填写页（ProfileSetupView）

---

## TC-05：冷启动 - Keychain 无 Token（首次安装）

**前置条件**：全新安装或手动清空 Keychain

**步骤**：
1. 打开 App

**预期结果**：
- `syncStatusFromServer()` 发现 token 为 nil，不发网络请求
- `registrationStep` 直接转为 `.login`
- 展示登录页

---

## TC-06：冷启动 - Token 已过期（服务端返回 401）

**前置条件**：Keychain 中存有过期或伪造的 token

**步骤**：
1. 打开 App

**预期结果**：
- `GET /api/v1/users/me` 返回 401
- 客户端捕获 `APIError`，清除 Keychain 中的 token
- `registrationStep` 转为 `.login`，展示登录页
- 不崩溃，不卡在 Loading 界面

---

## TC-07：冷启动 - 网络不可用

**前置条件**：Keychain 中存有 token，UserDefaults 中有上次状态（如 `active`），关闭网络

**步骤**：
1. 打开 App

**预期结果**：
- `GET /api/v1/users/me` 请求超时（15 秒后）或立即失败
- 客户端降级：使用 UserDefaults 中缓存的上次状态（`.active`）
- 直接进主页，不卡在 Loading 界面
- 不展示报错弹窗（静默降级）

---

## TC-08：RootView `.unknown` Loading UI 展示

**前置条件**：可通过调试断点或 slow network 拉长请求时间

**步骤**：
1. 打开 App，在 `syncStatusFromServer` 发出请求后、收到响应前观察 UI

**预期结果**：
- 背景色为 `#FAFAFA`（米白）
- 中心展示 `ProgressView` 菊花，tint 为 `#B5C0D0`（蓝灰）
- 无任何登录页或主页内容闪现

---

## TC-09：Token 写入和读取 Keychain 正确性

**步骤**：
1. 触发登录，观察 token 存储位置
2. 使用 Keychain 查看工具（如 Keychain Access 或调试日志）确认 token 存在
3. 强退重启，确认 token 可正常读取

**预期结果**：
- Token 仅存于 Keychain（`kSecClassGenericPassword`），UserDefaults 中无 `OB_AccessToken` 键
- 重启后 `KeychainHelper.read(key:)` 返回正确 token 字符串，非 nil

---

## TC-10：退出登录清除 Keychain

**前置条件**：已登录状态

**步骤**：
1. 调用 `authManager.logout()`（后续版本通过 UI 触发）

**预期结果**：
- Keychain 中 token 被删除
- `accessToken` 属性返回 nil
- `registrationStep` 转为 `.login`

---

## TC-11：并发 - 快速连续触发 mock 登录

**步骤**：
1. 快速点击登录按钮 3 次

**预期结果**：
- 仅发出一次网络请求（isLoading 为 true 时按钮不可交互或忽略重复触发）
- Token 不被重复写入或覆盖为错误值

---

## TC-12：Profile 提交 - token 从 Keychain 正确读取

**前置条件**：处于 `PENDING_PROFILE` 状态，Keychain 中有 token

**步骤**：
1. 填写昵称，点击完成

**预期结果**：
- `POST /api/v1/users/me/profile` 请求 Header 中携带正确的 `Authorization: Bearer <token>`
- 服务端返回 200，跳转至好友引导页

## 通用检查项

| 检查点 | 预期 |
|--------|------|
| 所有请求 Host | `ob-production-8afc.up.railway.app` |
| 所有请求路径前缀 | `/api/v1/` |
| Token 存储位置 | Keychain（非 UserDefaults） |
| mock code 格式 | `ob_dev_<UUID>`（每次唯一） |
| Loading 状态背景色 | `#FAFAFA` |
| 网络超时时间 | 15 秒 |
