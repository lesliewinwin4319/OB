## 1. Test Strategy Summary

测试重点在于 **导航架构的 Tab 切换与滑动切换的手势冲突隔离**（相机聚焦 vs 水平滑动、朋友圈垂直滚动 vs 水平切换）、**信息流的下拉刷新 / 到底提示 / Tab 重击刷新**、**单条 Feed 卡片布局与数据显示正确性**、**点赞交互的双入口（按钮 + 双击）与防抖机制**、**大图浏览器的缩放手势与退出方式**、**他人页跳转占位**、**TopNavigationBar 深色/浅色模式切换**、**BottomTabBar 半透明底与选中样式**，以及**快速连续操作、斜向滑动、极端图片比例**等边界场景。全部用例需真机手动执行。

## 3. Potential Risks

| # | 风险点 | 说明 | 建议 |
|---|--------|------|------|
| R1 | UIPageViewController 透明页手势穿透 | Page 0 需要 isUserInteractionEnabled=false 让触摸穿透到相机对焦，但 UIPageViewController 的滑动手势也需要工作，可能存在边界冲突 | 重点测试 TC-010、TC-011，关注相机页对焦与滑动切换的隔离 |
| R2 | 斜向滑动导致双手势同时响应 | 朋友圈页垂直滚动和水平切换在斜向滑动时可能同时触发 | TC-014 重点验证，若出现抖动需调整手势 delegate 的方向判断阈值 |
| R3 | 单击 200ms 延迟体验问题 | 系统需要等双击判定超时才确认单击，用户进入大图时可能感知延迟 | 这是系统行为，Instagram 等主流 App 也有相同延迟，属于可接受范围 |
| R4 | picsum.photos 图片加载不稳定 | Mock 图片来自外部网络服务，测试时可能因网络波动导致图片加载失败 | 首次测试确保联网环境良好；TC-073 专门覆盖图片加载失败的兜底表现 |
| R5 | NavigationStack 上移后拍照流程回归 | HomeView 重构为 CameraView，NavigationPath 上移到 MainTabView，可能影响拍照确认/上传选择流程 | TC-081 验证朋友圈与拍照流程互不干扰；需同时回归拍照流程核心路径 |
| R6 | 点赞防抖与 UI 即时更新的一致性 | 技术方案中提到 UI 即时变化 + 网络请求防抖，但 Mock 阶段防抖是否真正生效需验证 | TC-041、TC-042 验证快速操作下的数值一致性 |

## 4. Blocker 定义

以下情况属于该模块的阻塞级问题（Blocker），发现后必须修复才能继续：

1. **Tab 切换完全无法工作** -- 点击或滑动无法在相机和朋友圈之间切换
2. **朋友圈列表完全空白** -- 切换到朋友圈 Tab 后无任何 Feed 数据显示（Mock 数据未加载）
3. **App 崩溃** -- 任何操作导致 App crash（尤其是手势冲突、快速操作场景）
4. **相机功能被破坏** -- 因为导航架构改造导致相机预览黑屏或拍照流程无法进入
5. **手势完全冲突** -- 相机页无法对焦（被滑动手势拦截）或朋友圈无法垂直滚动（被水平切换拦截）
