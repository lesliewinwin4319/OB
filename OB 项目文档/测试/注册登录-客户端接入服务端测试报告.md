## Bug 详情

### BUG-REGL-001

- **Bug ID**：BUG-REGL-001
- **严重等级**：Major
- **所属模块**：客户端（Louis）
- **标题**：RootView `.unknown` Loading 状态实际不可达，TC-08 所描述的 Loading UI 永远不展示

**问题描述**：

`AuthStateManager` 的 `@Published var registrationStep` 默认值为 `.unknown`，但 `init()` 中立即调用了 `restore()`。`restore()` 会从 UserDefaults 读取上次状态并覆盖，若无记录则设为 `.login`。

SwiftUI 的 `@StateObject` 在视图首次渲染前完成初始化，因此当 `RootView.body` 第一次执行 `switch authManager.registrationStep` 时，状态已经是 `.login` 或 `.active`/`.pendingProfile`，永远不是 `.unknown`，`loadingView` 对应的 `case .unknown` 代码从未被执行到。

**代码路径**：
```
AuthStateManager.init()
  └── restore()
        ├── UserDefaults 有记录 → registrationStep = active / pendingProfile / login
        └── UserDefaults 无记录 → registrationStep = .login
```

**影响**：
1. TC-08（Loading UI 展示）无法按预期工作，测试用例需求与实现不符。
2. 首次安装的用户，冷启动时如果 Keychain 有 token 但 UserDefaults 无记录（Edge case：卸载重装），`restore()` 设为 `.login` 先展示登录页，`verifyTokenOnLaunch()` 后再跳到正确页面，可能出现 Login 页面闪烁。

**复现路径**：
1. 首次安装 App（UserDefaults 无 OB_RegistrationStep 记录）
2. Keychain 中存有有效 token（卸载重装场景：iOS 卸载不清 Keychain）
3. 打开 App，观察到短暂闪现登录页，再跳转到主页

**预期**：冷启动时先展示 `.unknown` Loading 界面，`verifyTokenOnLaunch()` 完成后再决定跳转目标

**实际**：`init()` 中 `restore()` 直接跳过 `.unknown` 状态，Loading UI 永远不可见

**修复建议**：将 `restore()` 从 `init()` 中移除，改为仅在 `verifyTokenOnLaunch()` 的网络失败降级分支中使用。`init()` 保持 `registrationStep = .unknown`，让 Loading UI 正常展示。

### BUG-REGL-002

- **Bug ID**：BUG-REGL-002
- **严重等级**：Major
- **所属模块**：客户端（Louis）
- **标题**：LoginView 登录按钮缺少 `isLoading` 禁用保护，快速连点可并发触发多次登录请求

**问题描述**：

`LoginView` 的 `wechatLoginButton` 没有 `.disabled(authManager.isLoading)` 修饰符。用户在登录请求进行中快速再次点击按钮，会再次调用 `handleWeChatLogin()` → `simulateWeChatAuthForDev()` → 发起新的 `loginWithMockCode()` Task。

虽然第二次请求的 token 会覆盖写入 Keychain（覆盖写是安全的），但这违反了 TC-11 的预期（"仅发出一次网络请求"），并且在弱网条件下可能造成两个不同 UID 的 token 竞争覆盖，导致最终存入 Keychain 的 token 属于非意图 UID。

**复现步骤**：
1. 打开 App，进入登录页
2. 快速连续点击「微信登录」按钮 3 次
3. 通过抓包或服务端日志观察，会发现 3 次 POST /auth/wechat/login 请求

**预期**：第一次点击后按钮变为禁用（`isLoading = true` 期间不可交互）

**实际**：按钮始终可点击，连续点击发出多次请求

**相关代码**：`LoginView.swift`，`wechatLoginButton` 定义处

**修复建议**：
```swift
private var wechatLoginButton: some View {
    Button(action: handleWeChatLogin) { ... }
        .disabled(authManager.isLoading)  // 添加此行
}
```

### BUG-REGL-003

- **Bug ID**：BUG-REGL-003
- **严重等级**：Minor
- **所属模块**：客户端（Louis）
- **标题**：Profile 提交成功后 `markProfileCompleted()` 未调用，本地状态未同步为 `.active`

**问题描述**：

`ProfileSetupView.handleNextStep()` 在 `submitProfile()` 成功后只调用了 `onProfileCompleted?()`（触发导航到 FriendGuideView），但没有调用 `authManager.markProfileCompleted()`，导致 `authManager.registrationStep` 仍停留在 `.pendingProfile`，UserDefaults 中的缓存也未更新为 `active`。

```swift
// handleNextStep() 成功分支（当前实现）
try await authManager.submitProfile(...)
onProfileCompleted?()          // 只做了导航
// markProfileCompleted() 从未在整个代码库中被调用
```

**影响场景**：

用户完成资料填写后跳转到 FriendGuideView，此时强退 App。
- 有网络：冷启动 → `verifyTokenOnLaunch()` → `GET /users/me` → 服务端返回 `ACTIVE` → 正确进主页（可恢复）
- 无网络：冷启动 → `verifyTokenOnLaunch()` 网络失败 → 降级读 UserDefaults → 读到 `pendingProfile` → 错误跳转到 ProfileSetupView，用户需要重新填写已提交过的资料

**严重等级判定为 Minor**：需要特定的时序（填写完成 + 进入好友引导页后 + 断网 + 强退），触发概率较低，且网络恢复后可自动纠正。

**修复建议**：在 `handleNextStep()` 成功分支加一行调用：
```swift
try await authManager.submitProfile(...)
authManager.markProfileCompleted()  // 添加此行，同步本地状态为 .active
onProfileCompleted?()
```

## 测试结论

**建议：修复后上线**

本轮共发现 3 个 Bug，无 Blocker。

- **BUG-REGL-001**（Major）：Loading UI 逻辑设计与代码实现存在根本性偏差，建议 Louis 调整 `AuthStateManager.init()` 初始化顺序，确保冷启动时 `.unknown` 状态可被渲染，防止卸载重装场景下登录页闪烁。
- **BUG-REGL-002**（Major）：并发防护缺失，登录按钮需加 `.disabled(authManager.isLoading)` 保护，改动量极小，建议同期修复。
- **BUG-REGL-003**（Minor）：Profile 提交后本地状态未更新，弱网强退场景下用户体验受损，建议修复。

服务端接口全部测试通过，字段映射、鉴权、参数校验逻辑均符合预期。客户端网络层、Keychain 存储、冷启动主流程（有网络时）逻辑正确，主流程可正常运行。
