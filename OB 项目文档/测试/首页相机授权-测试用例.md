## 1. Test Strategy Summary

测试重点在于**相机权限状态机的四种分支（notDetermined / authorized / denied / noHardware）正确驱动 UI**、**自定义权限弹窗的交互闭环（去设置 -> 返回 App 自动二次自检）**、**拍照按钮和导航入口的禁用/占位逻辑**、**Tap-to-Focus 交互细节**、以及**全局 ToastManager 的单例可靠性**。本模块为纯客户端功能，全部用例需真机手动执行。

## 3. Potential Risks

1. **sceneDidBecomeActive 延迟问题**：技术方案中提到从系统设置返回后延迟 0.3s 再检查权限，某些慢设备上 0.3s 可能不够，导致拿到旧的权限缓存值，UI 不更新。测试时需在旧设备上多次验证 TC-011 和 TC-013。

2. **AVCaptureSession 后台线程启动的 UI 同步**：session.startRunning() 在后台线程执行，但 @Published 属性的更新需要回到主线程。如果线程调度不当，可能出现 UI 更新延迟或紫色（主线程警告）崩溃。

3. **Tap-to-Focus 与 UI 手势冲突**：对焦手势（TapGesture）可能与导航入口、拍照按钮的点击手势冲突。需确认手势优先级：UI 按钮 > 对焦手势。

4. **权限弹窗弹出时机的竞态条件**：页面 onAppear 触发权限检查，如果 checkAndRequestPermission() 是异步的，弹窗弹出可能有短暂延迟。极端情况下用户可能在弹窗弹出前点击了其他 UI 元素。

5. **Toast 与权限弹窗同时存在**：如果用户在权限弹窗显示时点击了导航入口（弹窗背后的 UI），Toast 和弹窗可能同时存在，需确认视觉层级和交互逻辑。

6. **旧页面 Toast 与全局 ToastManager 的并存**：本期仅首页使用 ToastManager，但 LoginView、ProfileSetupView 等页面仍使用各自局部 Toast。如果用户在注册流程最后一步（好友引导页的局部 Toast）与首页（全局 Toast）快速切换，需确认两套 Toast 系统不冲突。

7. **内存泄漏风险**：CameraManager 持有 AVCaptureSession，作为 @StateObject 生命周期与 HomeView 绑定。首页作为 RootView 正常不会销毁，但退出登录等场景下如果 HomeView 被替换，需确认 session 正确释放。

8. **模拟器测试覆盖不足**：模拟器无真实摄像头（TC-046 noHardware），大量需要真实相机预览的用例（对焦、预览画面、震动反馈等）必须在真机上执行，增加了测试成本和周期。

## 4. Blocker 定义

以下情况属于阻塞级（P0），任意一项未通过则**不允许上线**：

| 编号 | Blocker 场景 |
|------|-------------|
| B-01 | 首次进入首页不弹出系统权限弹窗，用户无法授权相机（权限流程断裂） |
| B-02 | 用户授权相机后预览画面无法启动，停留在黑屏（核心功能不可用） |
| B-03 | denied 状态进入首页不弹出自定义权限引导弹窗（用户被困在黑屏无引导） |
| B-04 | 「去设置」按钮无法跳转系统设置页（权限补救通道断裂） |
| B-05 | 从系统设置开启权限返回 App 后 UI 不刷新，仍然黑屏（二次自检失效） |
| B-06 | 未授权时拍照按钮可点击且触发了拍照逻辑（产品明确要求禁用） |
| B-07 | 首页可侧滑返回到注册流程页面（导航约束失效） |
| B-08 | 三个导航入口点击无响应且无 Toast 提示（入口交互断裂） |
| B-09 | AVCaptureSession 启动失败时 App 崩溃（无异常兜底） |
| B-10 | 从好友引导页无法正常跳转至首页（注册 -> 首页衔接断裂） |
