## 方案概述

本方案描述 OB App 的注册登录后端实现思路：用户通过微信授权后，服务端会自动识别其是新用户还是老用户——新用户进入「资料填写」状态，老用户直接进入 App；整个过程基于微信唯一身份（OpenID）绑定账号，不会重复创建，且登录凭证（JWT Token）在资料填写阶段即已签发，服务端通过权限拦截确保未完善资料的用户只能访问注册相关接口。

## 一、数据库 Schema

### 1.1 用户表 `users`

```sql
CREATE TYPE user_status AS ENUM ('PENDING_PROFILE', 'ACTIVE');

CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    uid             VARCHAR(32)  NOT NULL UNIQUE,          -- 全局唯一用户标识，格式 ob_{snowflake}
    openid          VARCHAR(64)  NOT NULL UNIQUE,          -- 微信 OpenID，全局唯一
    unionid         VARCHAR(64),                           -- 微信 UnionID（预留，多端互通）
    nickname        VARCHAR(64),                           -- 用户昵称（预留可改）
    avatar_wx_url   TEXT,                                  -- 微信原始头像 URL
    avatar_cdn_url  TEXT,                                  -- CDN 长期头像 URL（异步任务写入）
    status          user_status  NOT NULL DEFAULT 'PENDING_PROFILE',
    created_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_users_openid  ON users(openid);
CREATE INDEX idx_users_uid     ON users(uid);
CREATE INDEX idx_users_status  ON users(status);
```

**字段说明：**
- `uid`：对外暴露的用户标识，隐藏自增主键；格式为 `ob_` + Snowflake ID
- `avatar_wx_url`：微信头像原始链接，有效期约 3 天，仅作临时备份
- `avatar_cdn_url`：异步任务下载后写入，正式对外使用的头像地址；本期为空时客户端降级使用 `avatar_wx_url`

---

### 1.2 头像异步任务表 `avatar_jobs`

```sql
CREATE TYPE avatar_job_status AS ENUM ('PENDING', 'PROCESSING', 'DONE', 'FAILED');

CREATE TABLE avatar_jobs (
    id          BIGSERIAL PRIMARY KEY,
    uid         VARCHAR(32)       NOT NULL,
    wx_url      TEXT              NOT NULL,
    cdn_url     TEXT,
    status      avatar_job_status NOT NULL DEFAULT 'PENDING',
    retry_count INT               NOT NULL DEFAULT 0,
    created_at  TIMESTAMPTZ       NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ       NOT NULL DEFAULT NOW(),

    CONSTRAINT fk_avatar_jobs_uid FOREIGN KEY (uid) REFERENCES users(uid)
);

CREATE INDEX idx_avatar_jobs_uid    ON avatar_jobs(uid);
CREATE INDEX idx_avatar_jobs_status ON avatar_jobs(status);
```

## 二、API 接口定义

### 2.1 微信登录 / 注册

**功能**：用微信 code 换取服务端 JWT，首次自动注册

```
POST /auth/wechat/login
Content-Type: application/json
Authorization: 不需要
```

**Request Body：**
```json
{
  "code": "wx_auth_code_from_miniprogram"
}
```

**Response 200（成功）：**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 604800,
  "user": {
    "uid": "ob_1234567890",
    "status": "PENDING_PROFILE",    // 或 "ACTIVE"
    "nickname": null,               // PENDING_PROFILE 时为 null
    "avatar_url": null              // PENDING_PROFILE 时为 null
  }
}
```

**客户端逻辑：**
- `status === "PENDING_PROFILE"` → 跳转资料填写页
- `status === "ACTIVE"` → 直接进入 App 主页

**错误码：**

| HTTP 状态码 | code | message |
|------------|------|---------|
| 400 | `INVALID_CODE` | 微信 code 无效或已过期 |
| 502 | `WECHAT_API_ERROR` | 微信服务端异常 |
| 500 | `INTERNAL_ERROR` | 服务器内部错误 |

---

### 2.2 完善用户资料（完成注册）

**功能**：PENDING_PROFILE 用户填写昵称和头像，状态流转为 ACTIVE

```
POST /users/me/profile
Content-Type: application/json
Authorization: Bearer {access_token}
```

**Request Body：**
```json
{
  "nickname": "陈小明",
  "avatar_wx_url": "https://thirdwx.qlogo.cn/mmopen/..."
}
```

**Response 200（成功）：**
```json
{
  "uid": "ob_1234567890",
  "status": "ACTIVE",
  "nickname": "陈小明",
  "avatar_url": "https://thirdwx.qlogo.cn/mmopen/..."   // 暂时返回微信 URL，CDN URL 异步写入
}
```

**错误码：**

| HTTP 状态码 | code | message |
|------------|------|---------|
| 400 | `INVALID_NICKNAME` | 昵称为空或长度超限（1-20字符）|
| 400 | `INVALID_AVATAR_URL` | 头像 URL 格式不合法 |
| 409 | `PROFILE_ALREADY_SET` | 用户已是 ACTIVE 状态，资料已存在 |
| 401 | `UNAUTHORIZED` | Token 无效或过期 |

---

### 2.3 获取当前用户信息

**功能**：客户端主动拉取用户状态（强退重进场景恢复页面）

```
GET /users/me
Authorization: Bearer {access_token}
```

**Response 200（成功）：**
```json
{
  "uid": "ob_1234567890",
  "status": "ACTIVE",
  "nickname": "陈小明",
  "avatar_url": "https://cdn.ob.app/avatars/ob_1234567890.jpg"
}
```

**说明：**
- `avatar_url` 优先返回 `avatar_cdn_url`，为空则降级返回 `avatar_wx_url`
- PENDING_PROFILE 用户也可访问此接口（Guard 放行）

**错误码：**

| HTTP 状态码 | code | message |
|------------|------|---------|
| 401 | `UNAUTHORIZED` | Token 无效或过期 |

---

### 2.4 获取好友申请列表（预留）

**功能**：预留接口，本期返回空列表

```
GET /users/me/friend-requests
Authorization: Bearer {access_token}
```

**Response 200：**
```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

**注意**：PENDING_PROFILE 用户访问此接口将被 Guard 拦截，返回 403。

## 三、核心业务逻辑

### 3.1 微信换 Token 流程（`POST /auth/wechat/login`）

```
客户端 code
    │
    ▼
[1] 调用微信 API：GET https://api.weixin.qq.com/sns/jscode2session
    参数：appid, secret, js_code, grant_type=authorization_code
    返回：{ openid, session_key, unionid? }
    │
    ├─ 失败 → 返回 502 WECHAT_API_ERROR
    │
    ▼
[2] 以 openid 查询 users 表
    │
    ├─ 不存在 → INSERT（幂等，见 3.2）→ status = PENDING_PROFILE
    │
    └─ 存在   → 读取现有记录         → status = 原有状态
    │
    ▼
[3] 签发 JWT
    payload: { sub: uid, status: user.status, iat, exp }
    有效期：7 天（PENDING_PROFILE 与 ACTIVE 一致）
    │
    ▼
[4] 返回 access_token + user 信息
```

**微信 session_key 处理**：本期不做微信数据解密功能，session_key 无需持久化存储。

---

### 3.2 UID 创建幂等（防并发重复注册）

并发场景下多个请求携带同一 openid 可能同时尝试创建用户，通过以下方式保证幂等：

```sql
INSERT INTO users (uid, openid, unionid, status)
VALUES ($1, $2, $3, 'PENDING_PROFILE')
ON CONFLICT (openid) DO UPDATE
    SET updated_at = NOW()   -- 无实质更新，仅触发 RETURNING
RETURNING *;
```

- `openid` 字段设有 `UNIQUE` 约束，数据库层面强保证唯一性
- `ON CONFLICT DO UPDATE` 确保并发时只有一条记录，且所有请求都能拿到同一个 uid
- UID 生成使用 Snowflake 算法（NestJS 服务层生成，格式 `ob_{id}`），在 INSERT 前生成，冲突时不会写入

---

### 3.3 状态机流转

```
                  ┌───────────────────────────────┐
                  │          状态说明               │
                  └───────────────────────────────┘

  [微信登录，首次]                    [完善资料接口]
       │                                   │
       ▼                                   ▼
  PENDING_PROFILE  ─────────────────►  ACTIVE
  （已授权，资料未完善）              （正常用户）

  Guard 规则：
  - PENDING_PROFILE：只允许访问 /auth/** 和 /users/me/profile 和 GET /users/me
  - ACTIVE：正常访问所有接口
  - 其余接口访问 → 403 Forbidden，body: { code: "PROFILE_INCOMPLETE" }
```

**Guard 实现思路（NestJS）：**

```typescript
// PendingProfileGuard
// 1. 从 JWT 解析 status 字段
// 2. 若 status === 'PENDING_PROFILE' 且当前路由不在白名单中
// 3. 抛出 ForbiddenException({ code: 'PROFILE_INCOMPLETE' })
```

白名单路由（装饰器 `@AllowPendingProfile()` 标记）：
- `POST /auth/wechat/login`
- `POST /users/me/profile`
- `GET /users/me`

### 3.4 CDN 头像异步任务设计

**目标**：将微信头像（有效期约 3 天）下载后上传至 OB CDN，获得永久 URL。

**本期：预留架构，不上线**

#### 触发时机
用户完成资料填写（`POST /users/me/profile`）时，服务端在写入 `users` 表后，往 `avatar_jobs` 表插入一条 PENDING 任务记录。

#### 任务处理流程

```
avatar_jobs 表（PENDING）
    │
    ▼
[Worker 轮询 或 BullMQ 队列消费]
    │
    ▼
[1] 下载 avatar_wx_url 图片二进制
    │
    ├─ 失败 → retry_count++，状态置 FAILED（重试上限 3 次）
    │
    ▼
[2] 上传至 CDN（阿里云 OSS / 腾讯云 COS，待运维确认）
    路径规则：avatars/{uid}/avatar_{timestamp}.jpg
    │
    ▼
[3] 更新 users.avatar_cdn_url = cdn_url
    更新 avatar_jobs.status = DONE，写入 cdn_url
```

#### 客户端兼容
- CDN URL 未就绪时，`GET /users/me` 返回 `avatar_wx_url` 作为降级
- CDN URL 就绪后自动替换，客户端无感知

#### 技术选型预留
- 任务队列：BullMQ（基于 Redis，与现有技术栈一致）
- Worker 模块：NestJS `@nestjs/bull` + 独立 Worker 进程
- 本期仅建表和插入任务逻辑，Worker 消费部分下期实现

## 四、关键技术风险及解决思路

### 风险 1：微信 code 被重复使用
**问题**：微信 code 只能使用一次，客户端重试可能导致 `40029 invalid code` 错误。  
**解决**：
- 服务端对微信 API 返回的错误码做精准区分（40029 = code 无效，返回 400；其他 = 微信服务异常，返回 502）
- 客户端在登录失败时重新拉取新 code，不复用旧 code

---

### 风险 2：并发注册导致重复用户
**问题**：同一用户极端网络下多次发起登录，可能触发并发 INSERT。  
**解决**：
- 数据库 `openid UNIQUE` 约束兜底，`ON CONFLICT` 保证最终只有一条记录
- 应用层加 Redis 分布式锁（key = `lock:register:{openid}`，TTL 5s）作为前置保护，减少无效 DB 写入

---

### 风险 3：JWT 中的 status 字段与数据库不一致
**问题**：用户完成资料填写后，旧 Token 中 status 仍为 PENDING_PROFILE，Guard 会持续拦截。  
**解决**：
- `POST /users/me/profile` 成功后，响应中不重新签发 Token（减少复杂度）
- Guard 对 status 敏感的接口，**不依赖 JWT 中的 status 字段**，而是**从数据库实时读取**用户状态
- JWT 中的 status 仅作为客户端路由跳转参考，服务端以 DB 为准
- 可配合 Redis 缓存（key = `user:status:{uid}`，TTL 5min）降低 DB 查询压力

---

### 风险 4：微信头像 URL 过期导致用户头像丢失
**问题**：微信头像 URL 有效期约 3 天，若 CDN 任务长时间未执行，头像将无法访问。  
**解决**：
- 异步任务设 SLA：任务入队后 10 分钟内完成处理
- 监控 `avatar_jobs` 表中 PENDING/FAILED 状态数量，超阈值告警
- FAILED 超过 3 次的任务进入死信队列，人工介入

---

### 风险 5：昵称长度与字符集边界
**问题**：emoji、中文、特殊字符的字节长度计算不一致。  
**解决**：
- 数据库字段 `VARCHAR(64)` 按 PostgreSQL 字符数（非字节数）计算，中英文各占 1 个字符
- 应用层校验：1 ≤ 字符数 ≤ 20（`[...nickname].length`，正确处理 emoji）
- 敏感词校验本期不做，字段结构已预留，后续接入时只需加 Validator 即可
