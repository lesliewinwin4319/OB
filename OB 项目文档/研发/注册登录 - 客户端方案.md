# 注册登录流程 · iOS 客户端技术方案

> **方案概述（写给产品负责人）**
> 整个注册登录流程由三个页面串联而成：登录页、资料填写页、好友引导页。用户通过微信一键授权后，App 会自动判断他的账号状态——如果资料还没填完，就回到填写页继续；如果已完成，就直接进首页。这套流程的核心设计原则是"状态驱动导航"：App 始终以服务端返回的用户状态为准，确保强退重进后用户不会迷路。

## 二、状态管理方案

### 2.1 全局状态：AuthStateManager（ObservableObject）

使用 `@StateObject` 在 `App` 入口注入，通过 `.environmentObject` 向下传递。

```swift
// 用户注册阶段枚举
enum RegistrationStep {
    case unknown            // 初始态，尚未确认
    case login              // 未登录
    case pendingProfile     // 已微信授权，待填写资料
    case active             // 资料已完善，可进首页
}

class AuthStateManager: ObservableObject {
    @Published var registrationStep: RegistrationStep = .unknown
    @Published var wechatUserInfo: WeChatUserInfo? = nil  // 微信授权返回的预填信息

    // 持久化：将 registrationStep 写入 UserDefaults，用于强退恢复
    func persist()
    func restore()

    // 服务端状态同步
    func syncFromServer(status: String)  // 接收 PENDING_PROFILE / ACTIVE
}
```

### 2.2 页面级状态：ProfileSetupViewModel

```swift
class ProfileSetupViewModel: ObservableObject {
    @Published var nickname: String = ""          // 绑定输入框
    @Published var avatarBgColor: Color           // 进入时随机固定
    @Published var showLengthToast: Bool = false  // 截断 Toast 开关

    var avatarChar: String { ... }  // 根据 nickname 计算动态字符
    var isNextEnabled: Bool { !nickname.isEmpty }
}
```

### 2.3 导航逻辑

使用 `NavigationStack` + `navigationDestination` 管理页面跳转。根视图监听 `AuthStateManager.registrationStep` 变化，决定展示哪个视图树：

```
.unknown  → 启动页 / Loading
.login    → LoginView
.pendingProfile → ProfileSetupView（在 AuthFlowView 内）
.active   → HomeView（脱离 AuthFlowView，替换整个根视图）
```

进入 `HomeView` 时，使用 `.navigationBarBackButtonHidden(true)` + 清空 NavigationStack 路径，彻底切断返回路径。

## 三、关键技术决策

### 3.1 微信 SDK 集成方式

- 通过 **CocoaPods** 引入 `WechatOpenSDK-XCFramework`（官方最新版本）
- 在 `AppDelegate` 中注册 AppID，实现 `WXApiDelegate`
- 授权结果通过 `onResp` 回调返回，拿到 `code` 后调用自有服务端换取 token
- 微信授权返回数据结构（`WeChatUserInfo`）：

```swift
struct WeChatUserInfo {
    let nickname: String?   // 可能为空或获取失败
    let openid: String
}
```

- **授权成功但资料获取失败处理**：`wechatUserInfo` 设为 `nil`，`nickname` 输入框保持空，正常跳转资料填写页，无预填。

### 3.2 WebView Sheet（用户协议/隐私政策）

使用 SwiftUI 原生 `.sheet` + `WKWebView` 包装：

```swift
// 封装组件
struct WebViewSheet: View {
    let url: URL
    @Binding var isPresented: Bool

    var body: some View {
        NavigationStack {
            WebViewRepresentable(url: url)  // UIViewRepresentable 包装 WKWebView
                .toolbar {
                    ToolbarItem(placement: .topBarTrailing) {
                        Button("关闭") { isPresented = false }
                    }
                }
        }
    }
}
```

URL 当前硬编码为百度（`https://www.baidu.com`），后续替换为实际协议/政策 URL 即可，无需改动结构。

### 3.3 禁止侧滑手势方案（好友引导页）

SwiftUI 目前无原生 API 直接禁用侧滑返回手势，采用以下方案：

```swift
// 方案：通过 UINavigationController 扩展禁用手势
struct FriendGuideView: View {
    var body: some View {
        content
            .navigationBarBackButtonHidden(true)  // 隐藏返回键
            .onAppear {
                // 获取当前 UINavigationController，禁用手势识别器
                if let navController = UIApplication.shared.topNavigationController {
                    navController.interactivePopGestureRecognizer?.isEnabled = false
                }
            }
            .onDisappear {
                // 离开时恢复（理论上不会触发，但作为保护）
                if let navController = UIApplication.shared.topNavigationController {
                    navController.interactivePopGestureRecognizer?.isEnabled = true
                }
            }
    }
}
```

由于好友引导页是注册流程末页，且进入首页后整个 NavigationStack 会被替换，禁用手势不会影响首页的正常交互。

### 3.4 30字符截断处理

在 `ProfileSetupViewModel` 中通过 `onChange` 监听输入：

```swift
// 在 View 绑定处
TextField("请输入昵称", text: $viewModel.nickname)
    .onChange(of: viewModel.nickname) { newValue in
        if newValue.count > 30 {
            // 截断到30字，取前30个 Character（正确处理 Emoji 等多字节字符）
            viewModel.nickname = String(newValue.prefix(30))
            viewModel.showLengthToast = true
        }
    }
```

注意：使用 `String.prefix(30)` 而非字节截断，确保 Emoji、中文等 Unicode 字符不会被切坏。Toast 展示 0.8 秒后自动消失，通过 `DispatchQueue.main.asyncAfter` 实现。

### 3.5 AvatarPreviewView 设计

动态头像字符提取逻辑：

```swift
func extractAvatarChar(from text: String) -> String {
    guard let first = text.unicodeScalars.first else { return "?" }

    // Emoji 判断：Unicode 扩展属性
    if first.properties.isEmoji && first.value > 0x238C {
        return String(text.unicodeScalars.prefix(1))
        // 注意：部分 Emoji 由多个 scalar 组成（如肤色修饰），需取完整第一个 Emoji
        // 使用 text.first 并转为 String 即可安全处理
    }

    let char = text.first!

    // 中文汉字范围：\u{4E00}–\u{9FFF}（基本汉字块）及扩展块
    if "\u{4E00}" <= char && char <= "\u{9FFF}" {
        return String(char)
    }

    // 英文字母
    if char.isLetter && char.isASCII {
        return String(char).uppercased()
    }

    // 数字
    if char.isNumber {
        return String(char)
    }

    // 其他（阿拉伯文、标点等）
    return "?"
}
```

头像背景色在 `ProfileSetupViewModel.init()` 中随机选取一次，存入 `@Published var avatarBgColor`，此后不再改变：

```swift
let morandi: [Color] = [
    Color(hex: "#B5C0D0"),
    Color(hex: "#CCD3CA"),
    Color(hex: "#F5E8DD"),
    Color(hex: "#EED3D9"),
]

init() {
    avatarBgColor = morandi.randomElement()!
}
```

字体颜色固定白色（`Color.white`），头像为圆形，使用 `.clipShape(Circle())`。

## 四、关键技术风险及解决思路

| 风险 | 描述 | 解决思路 |
|------|------|----------|
| **微信 SDK 回调时序** | `onResp` 在微信 App 返回时触发，可能在 App 进入前台过程中，此时 UI 还未完全 ready | 在 `onResp` 回调中先更新 ViewModel 状态，导航跳转延迟到 `onAppear` 或 `DispatchQueue.main.async` 中执行 |
| **微信未安装** | 用户设备未安装微信，`WXApi.isWXAppInstalled()` 返回 false | 点击登录按钮前检测，未安装则弹 Alert 提示「请先安装微信」，不调用 SDK |
| **NavigationStack 路径重置与动画冲突** | 从好友引导页跳首页时清空 NavigationStack，可能出现动画闪烁 | 使用 `withAnimation(.none)` 禁用该次跳转动画，或切换为 `fullScreenCover` 展示首页 |
| **Emoji 截断问题** | 用 `String.count` 计数时 Emoji 占 1 个字符，但用字节截断会切坏；部分组合 Emoji（如家庭 Emoji）由多个 scalar 组成 | 全程使用 `String.prefix(30)` 基于 `Character` 截断，避免切坏任何 Unicode 字符 |
| **强退恢复时服务端请求失败** | App 启动时调用服务端查状态，若网络异常无法确认 `registrationStep` | 本地 UserDefaults 缓存上次已知状态作为兜底，网络恢复后重新同步；Loading 期间展示启动图，避免页面闪烁 |
| **资料填写页键盘自动聚焦** | SwiftUI 的 `@FocusState` + `.onAppear` 聚焦有约 0.1s 延迟，可能用户感知到键盘"弹跳" | 在 `.onAppear` 中使用 `DispatchQueue.main.asyncAfter(deadline: .now() + 0.1)` 触发聚焦，与页面动画结束时机对齐 |
