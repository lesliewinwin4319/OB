## 1. 需求概要

用户完成注册流程后进入 App 核心首页。首页的核心是一个全屏后置摄像头预览画面，上面叠加导航入口（三个 Tab）和拍照按钮。首次进入时需要处理相机权限授权逻辑，权限被拒绝时展示自定义引导弹窗。

**一句话总结**：这是一个"相机预览 + 权限状态机 + UI 覆盖层"的组合页面，替换现有的 HomeView 占位视图。

## 2. 模块架构

整体采用分层结构，从底到顶依次为：

```
┌─────────────────────────────────────┐
│         HomeView（容器）              │  ← 替换现有占位 HomeView
│                                     │
│  ┌─────────────────────────────────┐│
│  │  CameraPreviewLayer（最底层）     ││  ← AVCaptureVideoPreviewLayer
│  │  或 纯黑背景（无权限时）           ││
│  ├─────────────────────────────────┤│
│  │  TopNavigationBar（顶部导航）     ││  ← 三个入口：我/朋友圈/朋友
│  ├─────────────────────────────────┤│
│  │  CaptureButton（底部拍照按钮）    ││  ← 本期仅占位，不执行拍照
│  ├─────────────────────────────────┤│
│  │  TapToFocusLayer（对焦交互层）    ││  ← 黄色方框 + 震动反馈
│  ├─────────────────────────────────┤│
│  │  PermissionDeniedAlert（弹窗）    ││  ← denied 时覆盖在最上层
│  │  ToastOverlay（全局 Toast）       ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
```

**设计原理（非技术语言）**：把首页想象成一叠透明卡片。最下面一张是相机画面，上面依次放导航栏、拍照按钮等 UI 元素。每张卡片各管各的逻辑，互不干扰，便于后续单独修改。

## 4. 核心类设计

### 4.1 CameraManager

```
CameraManager: ObservableObject
├── @Published permissionStatus: CameraPermission  // .notDetermined / .authorized / .denied / .noHardware
├── @Published isSessionRunning: Bool
├── @Published sessionError: String?
├── captureSession: AVCaptureSession              // 内部持有，不暴露给 View
│
├── checkAndRequestPermission()                    // onAppear 时调用
├── startSession()                                 // 权限通过后，在后台线程启动
├── stopSession()                                  // onDisappear 时调用
├── focusAt(point: CGPoint)                        // Tap-to-Focus，设置对焦点
└── recheckPermission()                            // 从设置页返回后二次自检
```

**权限枚举设计**：

```
enum CameraPermission {
    case notDetermined   // 未决定（首次）
    case authorized      // 已授权
    case denied          // 被拒绝（含 restricted）
    case noHardware      // 无摄像头硬件
}
```

**为什么单独定义枚举而不直接用系统的 AVAuthorizationStatus**：因为需要额外处理"无硬件"这个系统枚举不直接覆盖的场景，同时简化 View 层的 switch 分支。

### 4.2 CameraPreviewView（UIViewRepresentable）

```
CameraPreviewView: UIViewRepresentable
├── 输入：AVCaptureSession（从 CameraManager 获取）
├── 内部：创建 UIView，添加 AVCaptureVideoPreviewLayer 为 sublayer
├── videoGravity: .resizeAspectFill（全屏填充）
└── updateUIView: 监听 session 变化自动更新 layer
```

**为什么用 UIViewRepresentable 而不是纯 SwiftUI**：SwiftUI 没有原生的相机预览组件，必须桥接 UIKit 的 AVCaptureVideoPreviewLayer。这是 iOS 相机开发的标准做法。

### 4.3 ToastManager（全局单例）

```
ToastManager: ObservableObject（单例）
├── @Published currentToast: ToastItem?   // 当前展示的 Toast（nil = 不展示）
│
├── show(message: String, duration: TimeInterval = 1.5)
└── dismiss()
```

```
struct ToastItem {
    let id: UUID
    let message: String
}
```

**使用方式**：在 OBApp.swift 的 WindowGroup 根层注入 ToastManager，所有页面通过 `ToastManager.shared.show("xxx")` 调用，Toast 视图覆盖在 RootView 最上层。这样登录页、首页等所有页面都能使用同一个 Toast，不需要每个页面各写一份。

### 4.4 HomeView（容器视图）

```
HomeView: View
├── @StateObject cameraManager = CameraManager()
├── @EnvironmentObject authManager: AuthStateManager
│
├── body:
│   ZStack {
│       // Layer 1: 相机预览或黑屏
│       switch cameraManager.permissionStatus {
│           case .authorized:  CameraPreviewView + TapToFocusOverlay
│           default:           Color(hex: "#1A1A1A")  // 纯黑背景
│       }
│
│       // Layer 2: 顶部导航栏
│       TopNavigationBar()
│
│       // Layer 3: 底部拍照按钮
│       CaptureButton(isEnabled: cameraManager.permissionStatus == .authorized)
│
│       // Layer 4: 权限拒绝弹窗（条件展示）
│       if cameraManager.permissionStatus == .denied {
│           CameraPermissionAlert()
│       }
│   }
│
├── onAppear → cameraManager.checkAndRequestPermission()
├── onReceive(scenePhase) → .active 时 recheckPermission()
└── navigationBarBackButtonHidden(true) + 禁用侧滑
```

## 5. 关键流程

### 5.1 权限状态机流程

```
App 进入 HomeView
       │
       ▼
  onAppear: 调用 checkAndRequestPermission()
       │
       ├── 检查 AVCaptureDevice.default(.builtInWideAngleCamera) 是否为 nil
       │       │
       │       └── nil → permissionStatus = .noHardware → 黑屏（流程结束）
       │
       ├── AVCaptureDevice.authorizationStatus(for: .video)
       │       │
       │       ├── .notDetermined → 触发系统权限弹窗
       │       │       │
       │       │       ├── 用户允许 → permissionStatus = .authorized → startSession()
       │       │       └── 用户拒绝 → permissionStatus = .denied → 弹出自定义弹窗
       │       │
       │       ├── .authorized → permissionStatus = .authorized → startSession()
       │       │
       │       └── .denied / .restricted → permissionStatus = .denied → 弹出自定义弹窗
       │
       ▼
  用户去系统设置开启权限后返回 App
       │
       ▼
  sceneDidBecomeActive 触发 recheckPermission()
       │
       ├── .authorized → startSession()（弹窗自动消失）
       └── 仍然 .denied → 保持弹窗
```

### 5.2 Tap-to-Focus 流程

```
用户点击相机预览区域
       │
       ▼
  将 SwiftUI 坐标转换为 AVCaptureDevice 的焦点坐标（0~1 范围）
       │
       ▼
  调用 CameraManager.focusAt(point:)
       │
       ├── device.isFocusPointOfInterestSupported → 设置焦点
       ├── device.isExposurePointOfInterestSupported → 设置曝光点
       │
       ▼
  UI 反馈：
       ├── 在点击位置显示黄色方框（#E2C94C）
       ├── 方框从 1.0 缩小到 0.7，持续 0.3s
       ├── 缩小完成后淡出消失，持续 0.2s
       └── 触发 UIImpactFeedbackGenerator(.light) 轻震动
```

### 5.3 从注册流程进入首页的衔接

```
FriendGuideView 点击「进入 OB」
       │
       ▼
  authManager.transition(to: .active)
       │
       ▼
  RootView 监听到 registrationStep == .active
       │
       ▼
  切换到 HomeView（整个 AuthFlowView NavigationStack 被替换）
       │
       ▼
  HomeView.onAppear → 权限状态机启动
```

**说明**：现有 RootView 的路由逻辑完全不需要改动。`.active` 状态已经对应 HomeView，我们只是把占位内容替换为真正的相机首页。

## 8. 技术风险与应对

### 8.1 AVCaptureSession 启动耗时

**风险**：`session.startRunning()` 是同步阻塞调用，在主线程执行会导致 UI 卡顿。

**应对**：在专用的 `DispatchQueue(label: "com.ob.camera.session")` 后台线程启动和停止 session。启动完成后通过 `@Published isSessionRunning` 通知 UI 更新。

### 8.2 scenePhase 监听时机

**风险**：用户从系统设置返回 App 后，SwiftUI 的 `scenePhase` 变为 `.active`，但此时如果立刻检查权限可能拿到旧缓存值。

**应对**：在 `sceneDidBecomeActive` 回调中延迟 0.3s 再检查权限状态，确保系统更新完毕。使用 `@Environment(\.scenePhase)` 监听。

### 8.3 内存管理

**风险**：CameraManager 持有 AVCaptureSession，如果页面切换时未正确释放，会导致相机资源泄漏。

**应对**：在 HomeView 的 `onDisappear` 中调用 `stopSession()`。CameraManager 作为 `@StateObject` 生命周期与 HomeView 绑定，页面销毁时自动释放。但因为 HomeView 是首页，正常使用中不会销毁。

### 8.4 模拟器无相机

**风险**：Xcode 模拟器没有真实摄像头，`AVCaptureDevice.default` 返回 nil。

**应对**：CameraManager 检测到无硬件时设置 `permissionStatus = .noHardware`，UI 展示纯黑屏。开发调试时可加 `#if targetEnvironment(simulator)` 条件展示 mock 画面。

### 8.5 Toast 全局化的向后兼容

**风险**：现有三个页面（LoginView、ProfileSetupView、FriendGuideView）各自实现了局部 Toast，如果立刻全部迁移到 ToastManager，改动面较大。

**应对**：本次只在 HomeView 使用全局 ToastManager。现有页面的 Toast 保持不变，后续单独迭代统一迁移。

## 10. 开发顺序建议

按以下顺序逐步实施，每一步可独立验证：

1. **CameraManager + CameraPreviewView**：先跑通相机预览，确认真机能看到画面
2. **权限状态机**：处理 notDetermined / authorized / denied 三种分支
3. **HomeView 容器 + 黑屏兜底**：替换占位 HomeView，整合权限分支
4. **CameraPermissionAlert**：自定义权限弹窗
5. **CaptureButton + TopNavigationBar**：UI 覆盖层
6. **TapToFocusOverlay**：对焦交互
7. **ToastManager**：全局 Toast 基础设施
8. **Info.plist + 收尾测试**
