## 模块名称：拍照与共创提交流程

**功能概述**：用户在首页完成图像捕获后，对照片进行确认，并选择共创好友（本期单选）发起共创申请的完整业务流。

### 1. 详细逻辑说明

- **触发条件/入口**：首页状态下，点击底部的【拍照】快门按钮。
    
- **业务流程与状态同步**：
    
    1. **快门拦截与图像生成**：点击拍照后，App 立即截取当前 `AVCaptureSession` 的视频流帧，生成 `UIImage` 实例，并**立即推入（Push）**至「拍照确认页」。
        
    2. **异步本地保存与权限检查**：
        
        - 进入确认页的同时，后台线程静默触发相册保存动作。
            
        - 调用 `PHPhotoLibrary.requestAuthorization`。若为首次，系统弹窗询问相册权限；若用户拒绝，弹出 Toast：“请授权OB相册权限以保存照片”；若已授权，直接通过 `PHAssetChangeRequest` 将照片写入系统相册。
            
        - **关键点**：照片写入动作与用户在确认页的后续操作（无论是提交还是放弃）完全解耦，确保与微信体验一致的“拍了就存”。
            
    3. **好友选择器（Token 模式）**：
        
        - 在「上传选择页」，好友选择框与下方列表数据绑定。当选中一个好友时，通过状态机在顶部输入框渲染一个 Token（如 `@杰哥 [x]`）。
            
    4. **提交与状态机重置**：点击【申请】，向后端发起提交流程（携带照片 Data 和选中的 UserID）。完成后清理当前 Navigation Stack，重定向（Pop to Root）回首页。
        
- **异常分支 (Fallback)**：
    
    - **断网/弱网**：点击【申请】若 API 请求超时或失败，**不做本地持久化/草稿箱**，仅 Toast 提示：”网络不佳，请稍后重试”，用户需重新操作。（2026-04-08 产品确认：V1 不做草稿箱和自动重试）
        
    - **相册写入失败**：若因设备存储空间不足导致本地保存失败，不阻塞后续的共创申请流程，仅系统级 Log 记录。
        

### 2. UI/UX 规范

- **交互细节**：
    
    - **放弃拍照（二次确认）**：
        
        - 在「拍照确认页」点击左上角【×】。
            
        - 弹出 iOS 原生 `UIAlertController`（ActionSheet 样式或居中弹窗），Title: "确定要放弃吗？"
            
        - 操作项：`否` (Cancel Role，关闭弹窗)，`是` (Destructive Role，标红字样，点击后退回首页，此时本地相册中的照片**不删除**)。
            
    - **好友选择互斥逻辑**：
        
        - 点击下方列表的好友 Item 触发选中，Item 背景色变为 `#F5F5F5`（浅灰选中态）。
            
        - **单向取消限制**：再次点击列表中已选中的 Item **不会**触发取消操作。
            
        - **取消唯一路径**：必须点击上方选择框内 Token 右侧的 `[x]` 按钮（SF Symbol: `xmark.circle.fill`）才能移除选中状态，同时下方列表对应 Item 恢复未选中态。
            
    - **单选阈值拦截**：当已选中 1 名好友时，点击列表中其他未选中好友，拦截操作并触发 Toast：“只支持选择一名好友哦”。
        
- **按钮禁用与动态反馈**：
    
    - 【申请】按钮默认置灰（禁用状态，Alpha `0.3`）。
        
    - 仅当顶部选择框内存在 $\ge 1$ 个有效好友 Token 时，按钮高亮（Alpha `1.0`）并恢复可点击状态。
        
- **微信分享占位**：
    
    - 点击【微信分享】按钮，无实际跳转，仅底部弹出 Toast：“功能开发中，敬请期待”，持续时间 1.5s。
        
- **提交反馈**：
    
    - 点击【申请】成功后，退回首页，并弹出 Toast：“已提交！对方同意后展示在朋友圈”。
        

### 3. 开发建议 (Technical Tips)

- **数据结构防腐层设计（为多选做准备）**：
    
    - **强烈建议**：在 ViewModel 中，不要使用 `var selectedUser: User?` 来管理状态，而应直接声明为数组或集合：`@Published var selectedFriends: [User] = []`。
        
    - 本期的单选限制仅仅是在 UI 层的 `append` 操作前加一个 `if selectedFriends.count < 1` 的拦截器。这样未来扩展多选时，核心数据结构和 API 入参（如传 Array 给后端）完全不需要重构。
        
- **内存管理**：拍照生成的原始图片体积可能较大（10MB+），在 Navigation 传递 `UIImage` 时注意内存峰值。建议在展示预览图时，对图片进行适当的 Downsampling（降采样），原图仅保留在内存中用于最终的网络上传和本地保存。
    
- **路由管理**：建议使用 Coordinator 模式或 iOS 16+ 的 `NavigationPath` 管理路由状态。完成【申请】或【放弃】时，不要逐层 Pop，而是直接通过清空 Path 或重置 Root 状态回到首页，确保无闪烁的转场体验。


### 补充与修正逻辑说明

- **权限与核心流程绝对解耦 (Non-blocking Permission)**：
    
    - **逻辑修正**：保存照片至系统相册的动作，与 OB 的核心业务流（拍照 -> 确认 -> 选人 -> 申请）是**完全平行且解耦**的。
        
    - **处理机制**：无论用户对系统相册权限选择“允许”还是“拒绝”，**绝对不拦截**用户进程。若用户拒绝相册权限，仅抛出 Toast（“请授权OB相册权限以保存照片”），此时当前拍摄的图像流（`UIImage`）依然在 App 内存/沙盒中正常传递，确保用户能丝滑走完后续的 OB 内部共创闭环。
        
- **极端生命周期处理 (Process Kill / 中途退出)**：
    
    - **场景定义**：用户处于「拍照确认页」或「上传选择页」时，主动上滑杀掉 App 进程（Kill Process），或应用因后台资源紧张被系统强杀（OOM）。
        
    - **持久化策略（状态丢弃）**：在此阶段，捕获的照片数据和好友选中状态**仅驻留在内存 (RAM) 中**，不做本地磁盘持久化（CoreData/UserDefaults 不记录此中间态）。
        
    - **恢复逻辑**：一旦进程销毁，该次未完成的共创数据直接丢弃。用户下次冷启动 (Cold Start) App 时，路由状态机直接重置，**默认进入首页 (RootView - 相机实时预览)**，无需恢复草稿，不作任何中断提示。
